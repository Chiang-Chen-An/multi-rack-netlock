# Control Plane Plan

This document describes the control plane for the multi-rack NetLock prototype.
It is intended to be the implementation guide for the Python controller and the
contract between the controller, the leaf P4 pipeline, and the per-rack lock
server.

## 1. Goals

The control plane is responsible for the parts of NetLock that are too slow,
global, or stateful for the switch data plane:

1. Connect to every programmable switch through P4Runtime.
2. Install the P4 forwarding pipeline and baseline table entries.
3. Poll switch-resident lock state, queue state, and utilization counters.
4. Decide which locks should be handled in the switch and which should be
   handled by the lock server.
5. Coordinate safe lock migration between switch ownership and server ownership.
6. Allocate limited switch queue memory across hot locks.
7. Expose operational logs and metrics for experiments.

The data plane remains responsible for the fast path: parsing NetLock packets,
checking lock state, granting uncontended hot locks, buffering queued requests in
registers, forwarding cold locks to the lock server, and forwarding ordinary IP
traffic.

## 2. Current Code Layout

The current control-plane code lives in `control_plane/`:

| File | Role |
| --- | --- |
| `controller.py` | Top-level process, CLI parsing, controller lifecycle, switch connection orchestration. |
| `config.py` | JSON and CLI configuration model for switches and pipeline artifacts. |
| `p4runtime_switch.py` | Thin wrapper around `p4runtime-sh` for connecting to one P4Runtime switch and pushing the BMv2 pipeline. |
| `stats_collector.py` | Planned integration point for reading lock queue registers from a connected switch. |
| `migration_manager.py` | Migration state model and plan construction for switch-to-server and server-to-switch transitions. |
| `knapsack.py` | Memory allocation helper that chooses locks under a switch queue-slot budget. |
| `controller_config.example.json` | Example multi-switch-ready controller configuration. |

The current implementation fully covers configuration validation, P4Runtime
connection setup, pipeline push, and connection teardown. Register polling,
table programming for lock states, and lock-server coordination are the next
milestones.

## 3. Control-Plane Architecture

The intended runtime architecture is:

```text
+---------------------+
| controller.py       |
| main event loop     |
+----------+----------+
           |
           v
+---------------------+       +----------------------+
| P4RuntimeSwitch     | <---> | Leaf P4Runtime gRPC  |
| per switch handle   |       | server               |
+----------+----------+       +----------------------+
           |
           v
+---------------------+
| StatsCollector      |
| register snapshots  |
+----------+----------+
           |
           v
+---------------------+       +----------------------+
| Policy engine       | <---> | Lock server control  |
| hot/cold decisions  |       | channel              |
+----------+----------+       +----------------------+
           |
           v
+---------------------+
| MigrationManager    |
| transition safety   |
+---------------------+
```

The controller should be implemented as a single owner of switch control state.
Only the controller writes the P4 lock-state table and queue allocation
registers. The switch can update runtime queue registers on packet events, but
placement and migration decisions are control-plane owned.

## 4. Configuration Model

The controller supports two startup modes:

1. Single-switch CLI mode, using flags such as `--name`, `--device-id`,
   `--grpc-addr`, `--p4info`, and `--bmv2-json`.
2. JSON configuration mode, using `--config control_plane/controller_config.example.json`.

Each switch requires:

| Field | Meaning |
| --- | --- |
| `name` | Human-readable switch identifier, for example `leaf1`. |
| `device_id` | P4Runtime device ID used by the target. |
| `grpc_addr` | P4Runtime server address, for example `127.0.0.1:50001`. |
| `election_id` | P4Runtime controller election ID `[high, low]`. |
| `pipeline.p4info_path` | P4Info artifact generated by the P4 compiler. |
| `pipeline.bmv2_json_path` | BMv2 JSON artifact generated by the P4 compiler. |

Validation must happen before connection:

1. The controller must contain at least one switch.
2. `device_id` must be non-negative.
3. `grpc_addr` must be non-empty.
4. Pipeline artifact files must exist.
5. The same `(device_id, grpc_addr)` pair must not appear twice.

## 5. Switch Connection Lifecycle

For each configured switch, the controller performs this lifecycle:

1. Validate the switch and pipeline configuration.
2. Load `p4runtime_sh`.
3. Build a `FwdPipeConfig` from the P4Info and BMv2 JSON files.
4. Call `sh.setup(...)` with the device ID, gRPC address, election ID, and
   pipeline configuration.
5. Mark the switch as connected.
6. Run control tasks while the controller is active.
7. Call `sh.teardown()` on shutdown.

The existing `P4RuntimeSwitch` class is suitable for one-switch experiments.
`p4runtime-sh` stores process-global shell state, so a production multi-switch
controller should eventually replace it with raw P4Runtime gRPC client objects,
one client per switch.

## 6. P4 State Controlled by the Controller

The leaf data plane exposes the following control-relevant state.

### 6.1 Lock State Table

The `lock_state` table maps `hdr.netlock.lock_id` to one of these states:

| State | Value | Meaning |
| --- | ---: | --- |
| `HOT_FREE` | 1 | Lock is switch-owned and currently free. Acquire can be granted immediately. |
| `HOT_HELD` | 2 | Lock is switch-owned and currently held. Acquires may be queued in switch memory. |
| `COLD` | 3 | Lock is server-owned. Switch forwards acquire/release packets to the lock server. |
| `DRAINING` | 4 | Lock is moving from switch to server. New acquires go to the server while the switch drains existing holder/queue state. |
| `BUFFERING` | 5 | Lock is moving from server to switch. Switch can buffer new acquires while the server resolves old ownership. |

The controller is responsible for installing and updating this table. The data
plane reads the result on every NetLock packet.

### 6.2 Queue Allocation Registers

The switch uses one shared queue register pool. Each lock owns a fixed slice:

```text
absolute_slot = lock_queue_base[lock_id] + queue_offset
```

The per-lock queue metadata registers are:

| Register | Meaning |
| --- | --- |
| `lock_queue_base` | First shared queue slot owned by this lock. |
| `lock_queue_depth` | Number of queue slots allocated to this lock. |
| `lock_queue_head` | Circular offset of the next queued request to grant. |
| `lock_queue_tail` | Circular offset of the next enqueue position. |
| `lock_queue_occupancy` | Number of queued requests in this lock slice. |
| `lock_holder` | Tenant ID of the current lock holder, or `0` when free. |

The queued request payload registers are:

| Register | Meaning |
| --- | --- |
| `queue_tenant_id` | Tenant ID of the queued requester. |
| `queue_transaction_id` | Transaction ID of the queued requester. |
| `queue_priority` | Priority carried in the NetLock header. |
| `queue_src_mac` | Source MAC needed to return the grant. |
| `queue_src_ip` | Source IP needed to return the grant. |
| `queue_src_qp` | Source RDMA QP needed to return the grant. |
| `queue_ingress_port` | Ingress port used to send the grant back. |

The controller programs `lock_queue_base` and `lock_queue_depth` when allocating
memory to hot locks. It should reset `head`, `tail`, and `occupancy` only when it
is safe to prove the switch has no live queued requests for that lock.

## 7. Statistics Collection Plan

`StatsCollector.read_lock_queue_stats(lock_id)` should return a
`LockQueueStats` snapshot:

| Field | Source |
| --- | --- |
| `lock_id` | Controller input. |
| `base` | `lock_queue_base[lock_id]`. |
| `depth` | `lock_queue_depth[lock_id]`. |
| `head` | `lock_queue_head[lock_id]`. |
| `tail` | `lock_queue_tail[lock_id]`. |
| `occupancy` | `lock_queue_occupancy[lock_id]`. |
| `holder` | `lock_holder[lock_id]`. |

The stats collector should later be extended with:

1. Per-lock request counters from the switch.
2. Per-lock forwarded-to-server counters.
3. Per-lock queue-full counters.
4. Lock-server queue length and service-time statistics.
5. Migration completion and timeout counters.

These statistics feed the placement policy. The minimum useful policy input is
the request rate, queue occupancy, forwarded request rate, and current queue
depth for each lock.

## 8. Placement Policy

The controller decides which locks are switch-resident. A practical first policy
is:

1. Poll statistics every `poll_interval_ms`.
2. Estimate each lock's benefit from switch placement.
3. Estimate each lock's required queue slots.
4. Run `choose_locks_for_switch(candidates, slot_budget)`.
5. Compare the selected set against the current switch-resident set.
6. Migrate locks whose desired placement changed, subject to cooldown and safety
   constraints.

The existing `LockMemoryCandidate` model uses:

| Field | Meaning |
| --- | --- |
| `lock_id` | Candidate lock. |
| `value` | Estimated benefit from switch placement. Higher is better. |
| `slots` | Queue slots required by this lock. |

The existing `choose_locks_for_switch` function solves a 0/1 knapsack problem:
maximize total value while keeping total slots within the switch queue-slot
budget.

Suggested first value function:

```text
value = request_rate * server_avoidance_weight
      + queue_full_rate * overflow_penalty_weight
      + tail_latency_reduction_estimate
```

Suggested first slot estimate:

```text
slots = clamp(p95_queue_occupancy + burst_margin, min_slots_per_hot_lock, max_slots_per_lock)
```

The policy should avoid oscillation by enforcing:

1. Minimum residency time before a lock can migrate again.
2. Minimum value gap before replacing an existing hot lock.
3. Per-switch maximum migrations per control interval.

## 9. Migration Plan

Migration is the most important correctness boundary. The controller must not
lose queued requests, grant a lock twice, or split ownership between switch and
server.

### 9.1 Switch to Server

Switch-to-server migration moves a hot lock out of switch memory.

Initial state:

```text
HOT_FREE or HOT_HELD -> DRAINING -> COLD
```

Procedure:

1. Controller selects a switch-resident lock to evict.
2. Controller updates `lock_state[lock_id]` to `DRAINING`.
3. New acquire packets are forwarded to the lock server.
4. The switch continues handling the current holder and existing queued
   requests until the queue is empty.
5. Controller polls `lock_queue_occupancy[lock_id]` and `lock_holder[lock_id]`.
6. When `occupancy == 0` and `holder == 0`, the controller tells the lock server
   it now owns the lock.
7. Controller updates `lock_state[lock_id]` to `COLD`.
8. Controller may reclaim the queue slice for future allocations.

Safety rule: the controller must not clear queue registers or reassign this
lock's queue slice until the draining condition is observed.

### 9.2 Server to Switch

Server-to-switch migration moves a cold lock into switch memory.

Initial state:

```text
COLD -> BUFFERING -> HOT_FREE or HOT_HELD
```

Procedure:

1. Controller selects a server-owned lock to promote.
2. Controller allocates `queue_base` and `queue_depth` for the lock.
3. Controller initializes `lock_queue_head`, `lock_queue_tail`, and
   `lock_queue_occupancy` to `0`.
4. Controller updates `lock_state[lock_id]` to `BUFFERING`.
5. New acquires may be buffered in the switch if queue space is available.
6. Controller asks the lock server to transfer ownership state.
7. When the server reports no older outstanding holder or waiters, the
   controller sets:
   - `HOT_FREE` if no switch holder exists.
   - `HOT_HELD` if the switch has granted or imported a holder.
8. The switch becomes the owner for new acquire/release processing.

Safety rule: the server must stop granting new requests before ownership is
completed. During `BUFFERING`, forwarded and buffered requests must have a
well-defined ordering policy.

## 10. Controller Event Loop

The main loop should eventually follow this structure:

```text
connect switches
install baseline forwarding and lock-state entries
while running:
    poll switch stats
    poll lock-server stats
    update per-lock moving averages
    build memory candidates
    solve queue-slot allocation
    generate migration plans
    execute bounded number of migration steps
    emit logs and metrics
sleep until next interval
disconnect switches
```

The first implementation can run this loop synchronously. If lock-server RPCs or
multi-switch P4Runtime calls become slow, the controller should move to an
asyncio design with one task per switch and one policy task.

## 11. Table and Register Programming Requirements

The controller needs P4Runtime helpers for:

1. Writing `lock_state` table entries.
2. Reading register values by register name and index.
3. Writing register values by register name and index.
4. Installing baseline `ipv4_forward` entries.
5. Reading counter values once counters are added to the P4 program.

All table and register names used by the controller should be constants in one
Python module. This reduces mistakes when P4 object names change.

Every write should include structured logging:

```text
timestamp switch lock_id operation old_value new_value reason
```

For migration writes, include a migration ID so the full transition can be
reconstructed after an experiment.

## 12. Failure Handling

The controller should handle these failure modes explicitly:

| Failure | Required behavior |
| --- | --- |
| P4Runtime connection failure | Fail fast during startup if no required switch can be reached. |
| P4Runtime write failure | Stop the affected migration and mark it failed. Do not continue with later steps that assume the write succeeded. |
| Register read timeout | Keep the previous snapshot only if it is fresh enough; otherwise skip placement changes for that interval. |
| Lock-server control failure | Do not complete ownership transfer. Leave the lock in `DRAINING` or `BUFFERING` and retry or roll forward. |
| Controller crash | On restart, reconstruct state from switch registers, lock-state table entries, and lock-server ownership state before issuing new migrations. |

The safest default on uncertain ownership is to route the lock to the server
after the switch has drained, because the lock server can maintain larger and
more inspectable state than the switch.

## 13. Observability

The controller should write both human-readable logs and machine-readable
metrics.

Minimum logs:

1. Controller startup configuration.
2. Switch connect/disconnect events.
3. Pipeline push results.
4. Placement decisions.
5. Migration start, progress, completion, timeout, and failure.
6. P4Runtime errors.

Minimum metrics:

1. Per-lock request rate.
2. Per-lock switch queue occupancy.
3. Per-lock queue depth allocation.
4. Per-lock forwarded-to-server rate.
5. Number of switch-resident locks.
6. Migration count and duration.
7. Failed migration count.

These metrics should be exportable as CSV or JSON so the benchmark scripts can
plot throughput, latency, migration reaction time, fairness, and utilization.

## 15. Open Design Questions

The following questions should be resolved before the migration implementation
is considered complete:

1. What exact control protocol should the Python controller use to communicate
   with the lock server?
2. During `BUFFERING`, how should the system order requests already queued at
   the server against requests newly buffered in the switch?
3. Should `HOT_FREE` and `HOT_HELD` be controller-programmed states, or should
   the P4 program maintain them through data-plane table/register updates?
4. How should the controller recover the lock-state table after restart if the
   target does not support reading table entries reliably?
5. What minimum statistics are required to estimate lock value accurately under
   bursty workloads?

## 16. Development Rules

Control-plane code should follow these rules:

1. Keep P4Runtime access behind `P4RuntimeSwitch` or a future switch-client
   interface.
2. Keep policy decisions separate from low-level table/register writes.
3. Treat migration as a state machine, not a group of independent writes.
4. Unit test policy and migration logic without requiring BMv2.
5. Integration test P4Runtime operations against BMv2 once the P4 program builds.
6. Log every placement-changing write.
7. Prefer fail-stop behavior over silent partial migration.
