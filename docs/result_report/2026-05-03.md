# Test Result Report: 2026-05-03

## Summary

The current project snapshot was tested on the `p4-switch` VM. The P4 leaf
program compiles successfully with the VM's native `p4c-bm2-ss`, the compiled
BMv2 JSON loads into `simple_switch`, the Python unit tests pass, and the Python
controller can connect to `simple_switch_grpc` through P4Runtime and push the
configured forwarding pipeline.

Additional Mininet tests were run with the Scapy-based mock RDMA node. These
tests confirmed that the P4 dataplane statistic registers are incremented by
real packets in BMv2:

1. Cold-lock miss traffic increments `lock_acquire_count` and `lock_miss_count`.
2. Full-queue hot-lock traffic increments `lock_acquire_count` and
   `lock_overflow_count`.

The controller now supports a BMv2 Thrift-only telemetry mode for this VM. This
is required because BMv2 reports P4Runtime register reads as unimplemented, and
the installed `p4runtime_sh.shell` module does not expose a `RegisterEntry`
helper. The controller can read the same registers through `simple_switch_CLI`
without opening P4Runtime, which avoids clearing runtime register state.

One integration issue was found and fixed during testing: the VM's P4Runtime
shell exposes the controller API through `p4runtime_sh.shell`, not directly from
the top-level `p4runtime_sh` package. `control_plane/p4runtime_switch.py` was
updated to support this real package layout.

## Test 7: Mininet Cold-Lock Miss Statistic Test

Goal: verify that mock RDMA acquire packets increment miss-related statistic
registers in the P4 dataplane.

Topology:

```text
h1 -- s1(simple_switch_grpc) -- h2
```

Setup:

1. Mininet created hosts `h1`, `h2`, and `s1`.
2. `simple_switch_grpc` ran inside the `s1` namespace with:
   - `s1-eth1` as port `1`
   - `s1-eth2` as port `2`
3. The lock-state table was programmed through BMv2 Thrift:

```text
table_add NetlockIngress.lock_state NetlockIngress.mark_lock_state 1 => 3
```

This sets `lock_id=1` to `COLD`.

Traffic:

```text
3 mock RDMA acquire packets from h1, lock_id=1
```

```text
client/mock/rdma_node.py
```

Result: passed.

```text

lock_miss_count[1]= 3
lock_overflow_count[1]= 0
lock_queue_depth[1]= 0
lock_queue_occupancy[1]= 0
```

Derived metrics:

```text
MemoryWaste = 0 - 0 = 0 slots
MissRatio = 3 / 3 = 1.0
QueueOverflow = 0 / 3 = 0.0
```

Interpretation: the mock RDMA packets reached the P4 pipeline, matched the COLD
lock-state branch, and incremented both acquire and miss counters correctly.

## Issue Found and Fixed

### Problem

The first controller connection attempt failed with:

```text
AttributeError: module 'p4runtime_sh' has no attribute 'FwdPipeConfig'
```

The VM's P4Runtime shell repository has the shell API under:

```python
import p4runtime_sh.shell as sh
```

The project wrapper previously used:

```python
import p4runtime_sh as sh
```

### Fix

`control_plane/p4runtime_switch.py` now:

1. Imports `p4runtime_sh`.
2. Uses it directly if it exposes `FwdPipeConfig`.
3. Otherwise falls back to `p4runtime_sh.shell`.
4. Raises a clearer dependency error if the shell module or its dependencies are
   missing.

A unit test was added to cover the real submodule layout.

## Limitations of This Test Run

1. P4Runtime register reads are still unsupported by the current BMv2/P4Runtime
   shell environment.
2. Controller telemetry on this VM depends on BMv2 Thrift
   (`simple_switch_CLI`) and should be treated as a development fallback, not a
   portable target-independent telemetry API.
3. The Mininet tests covered acquire miss and acquire overflow only; release,
   dequeue, `DRAINING`, and `BUFFERING` packet paths were not tested.
4. No real RDMA verbs traffic was used; the traffic came from the Scapy-based
   mock RDMA node.
5. Lock-server migration coordination was not tested because the lock-server
   control path is not implemented in this snapshot.

## Overall Result

The current progress is healthy for the connection-management milestone:

1. P4 source compiles.
2. BMv2 accepts the compiled leaf pipeline.
3. Python unit tests pass.
4. Controller configuration validates.
5. The controller successfully connects to `simple_switch_grpc` on the
   `p4-switch` VM and pushes the pipeline.
6. Mininet plus mock RDMA traffic can increment the P4 statistic registers for
   cold-lock misses and queue overflow.
7. The controller can now collect those statistic registers using
   `--telemetry-thrift-only` without resetting BMv2 runtime state.

The remaining portability issue is target-independent register telemetry:
P4Runtime register reads are not available in this BMv2 setup, so production or
non-BMv2 environments should use a supported P4Runtime telemetry mechanism,
direct counters, digests, or a target-specific API.
